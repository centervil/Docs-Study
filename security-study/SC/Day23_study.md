## 📖 解説パート

### クラウドセキュリティの基本と責任共有モデル

クラウドサービスを利用する際、セキュリティに関する責任は、クラウドプロバイダーと利用者（顧客）の間で分担されます。この分担の考え方を「責任共有モデル」と呼びます。クラウドプロバイダーは、クラウド自体のインフラストラクチャ（ハードウェア、ネットワーク、物理的セキュリティなど）のセキュリティを担保しますが、顧客は、そのクラウド上で展開するアプリケーション、データ、アクセス権限、OSの設定など、顧客が管理する範囲のセキュリティについて責任を負います。

この責任範囲は、利用するクラウドサービスの形態（IaaS, PaaS, SaaS）によっても異なります。IaaS（Infrastructure as a Service）では、顧客はOS以上のレイヤーを管理する必要があるため、責任範囲は広くなります。PaaS（Platform as a Service）では、OSやミドルウェアの管理はプロバイダーが行いますが、アプリケーションやデータは顧客の責任となります。SaaS（Software as a Service）では、プロバイダーがほとんどのレイヤーを管理しますが、顧客はデータやアクセス権限の管理、利用方法といった部分で責任を負います。

責任共有モデルを正しく理解し、自社の責任範囲におけるセキュリティ対策を怠らないことが、クラウド環境における情報セキュリティ確保の第一歩となります。

### 主要クラウドプロバイダーのセキュリティ設計ポイント

#### Amazon Web Services (AWS)

AWSでは、責任共有モデルに基づき、プロバイダーは「クラウドのセキュリティ」を、顧客は「クラウド内のセキュリティ」を担当します。AWSのセキュリティ設計においては、IAM（Identity and Access Management）によるきめ細やかなアクセス制御、VPC（Virtual Private Cloud）によるネットワーク分離、Security GroupsやNetwork ACLsによるファイアウォール設定、KMS（Key Management Service）による暗号化キー管理が重要となります。また、AWS CloudTrailによる操作ログの記録と監視、Amazon GuardDutyによる脅威検出サービスの活用も推奨されます。

#### Microsoft Azure

Azureにおいても、責任共有モデルはAWSと同様の考え方です。Azureでは、Azure Active Directory（Azure AD）によるID管理とアクセス制御、Virtual Networkによるネットワーク分離、Network Security Groups (NSG) によるファイアウォール機能、Azure Key Vaultによる秘密情報・証明書・暗号化キーの管理が中心となります。Azure Security Centerによるセキュリティ体制の管理・脅威保護、Azure Monitorによるログ収集・分析も重要な要素です。

#### Google Cloud Platform (GCP)

GCPでも同様に責任共有モデルが適用されます。GCPでは、IAMによるアクセス権限管理、VPCネットワークによるセキュアなネットワーク構成、ファイアウォールルールによる通信制御が基本です。Cloud StorageやCompute Engineなどのサービスごとに、適切な暗号化設定を行うことが求められます。また、Cloud Loggingによるログ管理、Security Command Centerによるセキュリティ体制の可視化とリスク管理が提供されています。

#### 重要ポイント

*   **責任共有モデルの理解**: クラウドプロバイダーと顧客の責任範囲を明確に理解する。
*   **ID・アクセス管理**: IAM、Azure AD、GCP IAMなどを活用し、最小権限の原則に基づいたアクセス制御を行う。
*   **ネットワークセキュリティ**: VPC、仮想ネットワーク、ファイアウォールルールなどを適切に設定し、不正アクセスを防ぐ。
*   **データ保護**: 暗号化、キー管理サービス（KMS, Key Vault）などを利用し、データの機密性を確保する。
*   **監視・ログ管理**: CloudTrail, Azure Monitor, Cloud Loggingなどを活用し、不正行為の早期発見とインシデント対応に備える。

## 🏢 ケーススタディ

### ケース：クラウドストレージへの不正アクセスによる情報漏洩

ある企業が、顧客データを保存するためにAWSのS3バケットを利用していました。しかし、バケットのアクセス権限設定に不備があり、インターネット上に公開された状態になっていました。その結果、悪意のある第三者によって機密性の高い顧客情報が不正にダウンロードされ、情報漏洩が発生しました。

#### 問題点

*   S3バケットのアクセス権限設定の不備（公開設定）
*   機密データに対する適切な暗号化が行われていなかった
*   ストレージへのアクセスログの監視が不十分だった

#### 対応策

*   S3バケットのアクセス権限を最小限にし、必要最低限のユーザーのみがアクセスできるように設定する。
*   機密データは、保存時および転送時に暗号化する。
*   S3バケットへのアクセスログを有効にし、定期的に監視・分析する体制を構築する。
*   AWS IAMポリシーを見直し、不正な権限付与がないか確認する。
*   定期的なセキュリティ監査を実施する。

#### ケースから学ぶ教訓

*   クラウドサービスの設定ミスは、重大なセキュリティインシデントに直結する。
*   「デフォルトで安全」ではないため、常にセキュリティ設定を意識し、確認する必要がある。
*   アクセス権限管理とログ監視は、クラウドセキュリティの根幹である。

## ✅ テスト回答・解説

### 問題1 正解：3

解説：責任共有モデルにおいて、クラウドプロバイダーは自身が提供するインフラストラクチャ（ハードウェア、ネットワーク、データセンターの物理的セキュリティなど）のセキュリティに責任を持ちます。顧客は、そのインフラストラクチャ上で展開・利用するシステムやデータ、アクセス権限などについて責任を負います。したがって、選択肢3が正解です。

### 問題2 正解：3

解説：AWS IAMは、AWSリソースへのアクセスを安全に制御するためのサービスです。ユーザー、グループ、ロールを作成し、それらに特定のAWSリソースに対する権限（認証と認可）を付与することで、誰が何にアクセスできるかを細かく管理します。選択肢1、2、4は、IAMの直接的な主な目的ではありません。

### 問題3 正解：3

解説：Azure Virtual Networkは、Azure内のリソース（仮想マシンなど）が相互に安全に通信するための、論理的に分離されたプライベートネットワーク空間を提供します。これにより、オンプレミスネットワークとの接続や、リソース間のセキュアな通信経路を構築できます。選択肢1、2、4は、それぞれ別のAzureサービスが担当する機能です。
